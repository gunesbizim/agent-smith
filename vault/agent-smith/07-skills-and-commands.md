# Skills & Commands

Back to [[index]]. Agent Smith generates two kinds of artifact into `.claude/`:
**slash commands** (`commands/as-*.md`, user-facing orchestrators) and **worker skills**
(`skills/**/SKILL.md`, run in fresh subagents).

## Generated slash commands (`.claude/commands/as-*.md`)

Emitted by `src/scaffold/commands.ts` from `templates/commands/`, with `{{VAR}}` resolved.

| Command | Role |
|---|---|
| `/as-backend` | Implement a backend task respecting the detected architecture, layering, role enforcement, pre-push gates |
| `/as-frontend` | Implement a frontend task in the detected framework/UI library/state/i18n |
| `/as-test` | Test orchestrator — dispatch `test-backend` / `test-frontend` in fresh subagents; dispatches **only the side(s) that changed** (frontend-only PR → frontend skill only; backend-only PR → backend skill only; both changed → both) |
| `/as-pr-review` | Thin dispatcher — detects which sides changed and dispatches `pr-review-backend` / `pr-review-frontend`; dispatches **only the side(s) that changed**; **Step 0 = the sentrux remediation playbook** |
| `/as-documentation` | Documentation orchestrator — dispatch `docs-backend` / `docs-frontend`; dispatches **only the side(s) that changed** |
| `/as-ship` | Gated-autonomous path from finished work to a green PR (includes conventional-commit + push; `/as-git` was removed) |
| `/as-caveman` | Toggle caveman compression |
| `/as-insights` | Read architecture/decisions and suggest improvements |
| `/as-handoff` | Handoff orchestrator — write a structured `HANDOFF.md` and hand remaining work to fresh-context subagents when the context window is crowded (~60%+) or on request |

> **`/as-git` removed.** The plain commit/push workflow was folded into `/as-ship`. The
> scaffolded command set is now 9 commands: `as-backend`, `as-frontend`, `as-test`,
> `as-pr-review`, `as-documentation`, `as-ship`, `as-caveman`, `as-insights`, `as-handoff`.

**Every one of the 9 command templates** carries an `## Execution discipline (smith-mode)`
block that points the model at the vendored smith-mode skill
([[#smith-mode — execution discipline]]) — write a numbered stage map, delegate independent
stages, verify each stage with a failable check, self-review before delivery — for any task
spanning multiple files, sources, or sessions.

## Worker skills (`.claude/skills/<name>/SKILL.md`)

Emitted by `src/scaffold/skills.ts`. Six are templated worker skills; the rest are verbatim
helper stubs.

**Templated worker skills** (also the set LLM-regenerated by `generateSkills` —
[[04-generation-and-install#LLM-authored skills]]):

- `pr-review-backend`, `pr-review-frontend`
- `test-backend`, `test-frontend`
- `docs-backend`, `docs-frontend`

**Helper stubs copied verbatim:** `gitnexus/{guide,exploring,impact-analysis,debugging,
refactoring,cli}` (6) and `git-memory/{search,debug,index,status}` (4) — these teach the model how
to drive those MCP servers.

**Plus `smith-mode`** — copied verbatim into `.claude/skills/smith-mode/` ([[#smith-mode — execution discipline]]).

**Plus `handoff`** — copied verbatim into `.claude/skills/handoff/` ([[#handoff — session continuity]]).

## handoff — session continuity

`handoff` (template at `templates/skills/handoff/SKILL.md`) captures a structured session
continuity snapshot and hands the remaining work to fresh-context subagents. It triggers when
**either** holds: roughly 60%+ of the context window is used with multi-step work still
remaining, or the user explicitly asks to hand off / checkpoint / write a `HANDOFF.md` / split
the rest across fresh agents. It explicitly does **not** trigger when the task is nearly done,
when the remainder is a single trivial step, or when context is comfortable and the work is
coherent in one head.

The skill runs a four-stage loop (itself a smith-mode stage map):

1. **Take inventory** from artifacts, not memory — `git status`, `git branch --show-current`,
   `git log --oneline -10`, best-effort `gh pr list --head <branch>`, and the original
   task/plan file — phrasing each open item as a discrete, independently-actionable subtask.
2. **Write `HANDOFF.md`** at the repo root (updating in place if one exists) with eight fixed
   sections in order: **Goal / Current state / Branches & PRs / Done / In-progress /
   Next steps / Verification commands / Risks & guardrails**. The failable check re-reads the
   file and confirms every section is present and each "Next steps" item is independently
   actionable.
3. **Delegate one fresh subagent per remaining subtask** via the **Agent** tool —
   `isolation: "worktree"` for subtasks that edit files, none for read-only ones; independent
   subtasks dispatched concurrently, dependent ones sequenced. The failable check confirms the
   number of subagents dispatched equals the number of open subtasks (nothing dropped or
   double-assigned).
4. **Collect, verify, self-critique** — fold finished items back into `HANDOFF.md` with their
   evidence, re-run the project's verification commands on the integrated result (a subagent's
   success report is a claim, not proof), and name at least one weakness before declaring done.

The skill carries **no `{{VAR}}` template variables** — like `smith-mode`, it is copied
verbatim (no substitution) and is stack-agnostic. `HANDOFF.md` is a working artifact a project
may choose to gitignore.

It pairs with the deterministic context hooks documented in
[[05-hooks-and-events]]: the **PreCompact** `pre-compact-handoff` hook writes a best-effort
snapshot before a context compaction, and a **UserPromptSubmit** nudge suggests running
`/as-handoff` at ~60% context. The skill notes that the hook's snapshot is a safety net, not a
substitute — the hook cannot reason about which work is done or delegate it, so the skill
should be run deliberately while there is still enough context to write a good handoff. The
`/as-handoff` slash command is the user-facing orchestrator that dispatches this skill in a
fresh subagent.

## smith-mode — execution discipline

`smith-mode` is an **execution-discipline skill** vendored from
[mrtooher/smith-mode](https://github.com/mrtooher/smith-mode) and bundled with agent-smith
(template at `templates/skills/smith-mode/{SKILL.md,EXAMPLE.md}`). It encodes a four-step loop
for complex work — **stage map → delegate → failable verification → self-critique** — and is
explicit that trivial single-pass tasks should skip it.

`src/scaffold/skills.ts` copies the whole `smith-mode` template dir **verbatim** (no `{{VAR}}`
substitution) into every project's `.claude/skills/smith-mode/` on init. Three layers then point
at it so it is actually read:

- **Every `/as-*` command** carries an `## Execution discipline (smith-mode)` block referencing
  `.claude/skills/smith-mode/SKILL.md`.
- The **SessionStart hook** surfaces it each session — see
  [[05-hooks-and-events#smith-mode additionalContext]].
- The **LLM skill generator** (`src/adapt/llm-skills.ts`) instructs generated skills to
  *reference* the smith-mode skill rather than duplicate its content.

## PR #65 — shipped behaviour changes

### /as-pr-review: thin dispatcher + adversarial critic panel in skills

**`/as-pr-review` is now a thin dispatcher.** It detects which sides of the stack changed and
dispatches `pr-review-backend` and/or `pr-review-frontend` in fresh subagents. The orchestrator
also runs **Step 0** (the sentrux remediation playbook) if a quality regression is detected.

> **Side-selective dispatch (explicit rule):** `/as-pr-review`, `/as-test`, and
> `/as-documentation` each dispatch **only the side(s) that actually changed**:
> - Frontend-only change → frontend skill only (`pr-review-frontend` / `test-frontend` / `docs-frontend`)
> - Backend-only change → backend skill only (`pr-review-backend` / `test-backend` / `docs-backend`)
> - Both sides changed → both skills, each in its own fresh subagent
>
> Running both skills on a single-side PR wastes tokens and invents findings. The orchestrators
> are expected to detect the change surface before dispatching.

**The adversarial critic panel was moved into the skills.** `pr-review-backend` and
`pr-review-frontend` now each embed the five-lens critic panel scoped to their side:

| Lens | Skill |
|---|---|
| Security | `pr-critic-security` |
| Performance | `pr-critic-performance` |
| Simplicity | `pr-critic-simplicity` |
| Maintainability | `pr-critic-maintainability` |
| Developer experience | `pr-critic-dx` |

Each lens runs as a fresh subagent and tries to **refute** the change from its standpoint
(adversarial — looking for reasons to reject, not approve). Critics also use MCP tools: gitnexus
(impact/blast-radius, callers) and sentrux (quality signal, coupling, cycles) per lens. The skill
synthesizes findings across lenses using a **four-tier severity model** (critical / high / medium /
low) and a **false-positive gate**:

1. Each critic emits findings tagged with severity and confidence.
2. The skill **drops confirmed FPs** in synthesis, then:
   - **Auto-fixes confirmed critical/high** findings (applied in the same turn, not queued).
   - **Leaves medium/low** as non-blocking follow-ups (listed but do not block merge).
   - **Escalates uncertain** findings to the human for a final call.
3. The **merge verdict blocks only on confirmed critical/high**.

### Subagent model routing

All `/as-*` orchestrators and worker skills now route subagents by phase type, mirroring
`src/engine/tdd-engine.ts → modelForPhase`:

| Phase type | Model |
|---|---|
| Exploration, planning, review | fresh **Opus** subagent |
| Implementation, execution | fresh **Sonnet** subagent |

Applies to: `as-test` / `as-pr-review` / `as-documentation` dispatch and
`as-backend` / `as-frontend` internal phases.

### as-backend / as-frontend: explore → triage → TDD-plan → RED-first

`/as-backend` and `/as-frontend` now open with a structured four-step approach before writing
any production code:

1. **Explore** (Opus) — read relevant code, map callers/dependents via gitnexus.
2. **Triage** — identify the minimal change surface.
3. **TDD-plan** (Opus) — write a test plan with ≥1 failing test per acceptance criterion.
4. **RED-first implement** (Sonnet) — write the failing tests first, then the code that makes
   them pass. A commit that adds tests without a prior red run is rejected.

### TDD everywhere: RED-first in all implementation and test skills

All implementation skills (`/as-backend`, `/as-frontend`) and test skills (`test-backend`,
`test-frontend`) carry an explicit **RED-first mandate**: a failing test covering the new
behaviour must be committed (and its failure proven in `red-proof.json`) **before** the
production code it covers is written or committed. This mirrors the engine's RED phase contract.

The RED-first rule is injected into the LLM skill-generator prompt
(`templates/prompts/skill-generator.md`) so it also survives the LLM generation path — generated
skills pick it up automatically. It is also written into `best-practices.md` (the **Followed**
bucket), so every skill that references the best-practices doc enforces it as a rule rather than
a suggestion.

### as-ship: branch hygiene + sentrux remediation loop

`/as-ship` now:
- Runs `decideBranch()` ([[09-pipeline#Pure pipeline helpers]]) as a preflight to ensure the
  working branch is forked from an up-to-date `origin/main` before pushing.
- Enters the bounded sentrux remediation loop on a regression (details in
  [[08-sentrux-quality-gate#Bounded remediation loop in /as-ship and /as-pr-review]]) rather
  than hard-stopping immediately.

## Customization — how a template becomes stack-specific

`src/adapt/skill-customizer.ts → customizeSkills()` runs two passes over each command/skill,
**customizations first, substitution last** (B8 — the reverse of the old order, which could
re-inject `{{VAR}}` after substitution and ship it unresolved):

1. **`applyFrameworkCustomizations()`** — remove sections that don't apply to the detected stack:
   - non-Django → strip `## Django / DRF patterns`, rewrite `python manage.py` →
     `{{BACKEND_MIGRATE_CMD}}`, `drf-spectacular` → `{{API_DOCS_LIBRARY}}`.
   - non-Vue → strip `<script setup>` blocks, rewrite `.vue` / `Vuetify 3`.
   - non-Vuetify → strip `## Vuetify`.
2. **`substituteToFixpoint()`** — the authoritative final pass: resolve `{{VAR}}` repeatedly until
   stable (so any placeholder the step-1 customizations re-injected is also resolved), then
   `console.warn` on any residual unknown `{{...}}` rather than shipping it silently.

> **Generator prompt templates (P1/C1):** the LLM skill generator's master prompt is itself a
> template surface — `templates/prompts/skill-generator.md` (+ `skill-stub-example.md`, inlined as
> the "decorate, don't replace" anchor). `loadSkillGeneratorPrompt()` reads them from the package
> root and interpolates `{{SKILL_LIST}}`/`{{STUB_EXAMPLE}}`; `AGENT_SMITH_PROMPTS_DIR` overrides
> the dir. See [[04-generation-and-install#LLM-authored skills]].
>
> **MCP-first grounding (#61).** The prompt now instructs the generator to **PREFER
> code-intelligence / documentation MCP tools when grabbing code or documentation**, in the
> order gitnexus (architecture map, call graphs, impact) → git-memory (why code changed) →
> documentation MCP (obsidian / context7 / similar), and to fall back to raw `Read` / `Glob` /
> `Grep` **only** when no MCP tool can answer or none are configured (silently skipping ones a
> project lacks). See [[04-generation-and-install#LLM-authored skills]] for the matching
> generator-side wiring (dynamic code-intel MCP allowlist, configurable
> `AGENT_SMITH_SKILLS_TIMEOUT_MS` timeout, dashboard tool/MCP visibility).

> **De-Django'd backend templates.** The backend templates (`docs-backend`,
> `pr-review-backend`, `test-backend` SKILL.md and the `as-documentation` command) no longer
> hardcode Django/DRF terms (`APIView`, `serializers`, `ruff`, `manage.py`,
> `drf-spectacular`). They are now framework-neutral and `{{BACKEND_FRAMEWORK}}`-driven —
> e.g. "request handlers" and "request/response schemas (DTOs/serializers)" instead of
> `APIView`/serializer. As a result none of the shipped templates currently contain a
> `## Django / DRF patterns` (or `<script setup>` / `## Vuetify`) heading; the
> `applyFrameworkCustomizations()` strip passes above remain in the code as a safety net but
> are effectively no-ops on the current templates. Framework-agnostic guidance therefore
> survives on every stack by default.

## Best practices in generated skills

Every worker skill references **`docs/architecture/best-practices.md`** and carries a
**"Recommended best practices"** section. The split is deliberate:

- **Followed** standards (what the project already does) → encoded as the skill's **enforced
  rules**; reviewers block on these.
- **Recommended** standards (good practices not yet adopted) → surfaced as **suggestions**
  (2–5 per skill), grounded in recognized engineering standards for the detected stack and the
  latest documentation available. They never block a merge on their own.

The stubs ship a curated, framework-neutral baseline; the LLM generator
([[04-generation-and-install#LLM-authored skills]]) refreshes the *Followed* bucket from the real
code and tailors *Recommended* to the actual stack. Backend stubs are now framework-neutral and
`{{BACKEND_FRAMEWORK}}`-driven outright (no Django/DRF hardcoding), and any remaining
stack-specific examples are labelled as examples, so the no-LLM fallback stays usable on any
stack.

## Documentation skills

`docs-backend` and `docs-frontend` (templates in `templates/skills/docs-*`) write project
documentation into the **Obsidian vault via the `obsidian` MCP**, falling back to in-repo `docs/`
when the MCP isn't connected. They run **inside the Claude Code session** — the Claude Code CLI
*is* the LLM, so generation is on by default with **no `--llm` flag, no API key, no external
model**. `docs-frontend` additionally drives the app with the **Playwright MCP** and captures
per-role screenshots into the gitignored `.playwright-mcp/`, and uses the **chrome-devtools MCP**
(console, network, performance) for real runtime observation — not just static screenshots.
The `smoke-test` skill similarly uses chrome-devtools (console errors, network failures, perf
insights) to verify a live app after a deploy or merge. `pr-review-frontend` and `test-frontend`
reference the wired UI-library MCP (`{{FRONTEND_UI_LIBRARY}}`, e.g. vuetify) for component API
lookups, rather than an unnamed "component library MCP". Orchestrated by `/as-documentation`,
which detects what changed on the branch and dispatches the matching skill(s). See
[[06-mcp-servers#Obsidian — special handling]].

## CLAUDE.md managed block

`src/adapt/claude-md-writer.ts → writeClaudeMd()` runs **last** in `init` (after every command
and skill is scaffolded) and writes/refreshes an **agent-smith-managed block** in the target
project's `CLAUDE.md`. It is **non-destructive**: agent-smith owns *only* the content between
the markers

```
<!-- agent-smith:start -->  …  <!-- agent-smith:end -->
```

— everything outside is the user's and preserved verbatim. On re-init the block is regenerated
in place; if no `CLAUDE.md` exists one is created with just the block; if a `CLAUDE.md` exists
without markers the block is appended below the user's content.

The block enumerates the **full current capability surface** via `buildManagedBlock()`:

- a `## Slash commands` table — scanned from `.claude/commands/*.md` (name = filename, purpose =
  first prose line),
- a `## Skills` table — scanned recursively from `.claude/skills/**/SKILL.md` (name + description
  from YAML frontmatter, folded `>` descriptions supported),
- a `## Execution structure` section — command → main skill → sub-skill → MCP tool,
- a `## Test-driven development (mandatory)` RED-first section (write failing test, prove red,
  then write the code),
- a `## Available MCP tools` table (gitnexus, git-memory, playwright, chrome-devtools, sentrux,
  obsidian) with a pointer to `docs/architecture/mcp-tools.md`,
- an intro line directing the model to follow **smith-mode** discipline (stage map → delegate →
  failable verification → self-critique) for any task spanning multiple files, sources, or
  sessions.

So an agent reading `CLAUDE.md` always sees every available `/as-*` command and skill plus the
smith-mode pointer, the execution structure, the TDD mandate, and the MCP tool surface. See also
[[04-generation-and-install#CLAUDE.md managed block]].
