The most important finding
The core promise is broken for the README's own flagship example. The README headline claims that on a Go/Echo project, "Pre-push gates become golangci-lint run + go test ./... instead of ruff + mypy + pytest." I built a Go/Echo + React/Zustand monorepo exactly as described and ran init. The generated .claude/commands/backend.md says, verbatim:


**Stack:** Go 1.25, Echo      ← header is correct
...
ruff check .                  ← Python linter, on a Go project
pytest -m 'not integration'   ← Python test runner, on a Go project
So the skill tells Claude the stack is Go but instructs it to run Python tooling. The root cause is in best-practice-mapper.ts: DEFAULT_TEMPLATE_VARS is hardcoded to a Django/Vue project, and the mapper only overrides BACKEND_TEST_CMD/BACKEND_LINT_CMD if test files and a linter config already exist. The format command, migrate command, and settings module are never language-corrected, so they stay ruff format --check ., python manage.py makemigrations, and config.settings.dev for every non-Python project. The analyze output confirmed it: a Go project reported BACKEND_TEST_CMD: pytest, BACKEND_LINT_CMD: ruff check ., BACKEND_MIGRATE_CMD: python manage.py makemigrations.
This is the single highest-leverage thing to fix. The architecture should be inverted: derive a complete set of language-appropriate command defaults from the detected language (a small per-language table: Go → go test ./.../golangci-lint run/gofmt; Rust → cargo test/clippy/cargo fmt; etc.), and treat the Django values as just one entry in that table rather than the global baseline.
Other concrete correctness bugs
The same run surfaced a cluster of accuracy problems, all worth fixing:
* Hardcoded versions contradict the "exact versions" claim. My go.mod said go 1.22; the detector reported go 1.25 because detectGoBackend hardcodes the string. There are two different hardcoded Go versions (1.22 in one path, 1.25 in another) — they should both parse the actual go X.Y directive. Likewise all TS frameworks are stamped 5.x.
* Self-contradicting ORM/DB output. The report said ORM: none detected and Database ORM: sqlc in the same output — because the Echo branch hardcodes orm: null while detectDatabase fabricates sqlc for any Go+pgx project. I never included sqlc. Guessed values ("postgresql", "sqlc", "JWT") are presented as detected facts and then written into skills.
* Missed library in the monorepo path. My go.mod had go.uber.org/zap, but the report said Logging: unstructured — the subdir detection path doesn't run the zap/logrus check the root path does.
* agent-smith --version reports 0.1.0 while package.json and the plugin manifest say 0.4.0 — cli/index.ts hardcodes the old string.
* Missing LICENSE file even though package.json's files array lists it and the README says MIT — npm will warn on publish.
* README documents npx agent-smith insights as a CLI command, but it isn't registered in cli/index.ts (only init/configure/analyze/doctor/ticket/pipeline exist).
One latent risk worth noting: applyFrameworkCustomizations runs after template substitution and can re-inject {{...}} placeholders (e.g. it replaces python manage.py with # {{BACKEND_MIGRATE_CMD}}). It didn't fire in my run because the templates don't contain that literal, but the ordering is fragile — substitution should run last, or run in a loop until stable.
Maintainability / architecture ideas
project-detector.ts is 1,263 lines of deeply repeated object literals — every framework hand-writes a near-identical BackendInfo. This is where most of the version/logging/ORM bugs live, because a fix has to be applied in many places. I'd refactor to a data-driven registry: one declarative table mapping { marker, framework, language, defaults }, with a single builder. That collapses the file dramatically and makes "detect the real version" a one-line change instead of twenty. The detectGoBackend/detectRustBackend functions are already near-duplicates of the inline root-path logic — classic sign the data should be unified.
There's also duplicated infrastructure: readJson, readFileSafe, and pkgDeps are reimplemented in both project-detector.ts and package-scanner.ts, and findSubPackages is re-walked by several functions. A shared fs-helpers module with a small cache for repeated directory walks would cut both duplication and redundant I/O. Relatedly, analyze doesn't call scanPackages (only init does), so analyze --json omits the rich version data — they should share one analysis function.
The honesty gap on the pipeline
The README sells an "autonomous ticket-to-PR pipeline" with vivid phase diagrams, but runPipeline and the ticket/pipeline commands are stubs that print success without doing anything. I don't think this is dishonest by intent — the real execution clearly happens inside Claude Code via the scaffolded skills, not via this CLI. But a new user running npx agent-smith ticket PROJ-123 --auto will see "PR created" with no PR. I'd either (a) clearly label these commands as experimental/not-yet-implemented in both --help and the README, or (b) make them genuinely orchestrate by shelling out to claude headlessly. Option (a) is a 20-minute honesty fix; (b) is the real roadmap item.
If I were prioritizing
The order I'd tackle these: first the language-default inversion in best-practice-mapper (it's the product's whole reason to exist, and it's wrong for everything except Django/Vue); then add a golden-output test that runs init against fixture repos for Go, Rust, NestJS, and Django and asserts the generated skills contain the right commands and no Python tooling — that single test would have caught the headline bug and guards every future change. After that: parse real versions, fix the version-string drift, add the LICENSE, reconcile the README's CLI list, and either label or implement the pipeline. The registry refactor and the fs-helper consolidation are the larger structural wins that make all the correctness fixes cheaper.
The foundation here is genuinely solid and the ambition is clear — the gap is that the detection layer's accuracy hasn't been validated end-to-end against the non-default stacks the README advertises.
